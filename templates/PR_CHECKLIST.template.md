# Pull Request Checklist

Use this checklist before requesting review.

## Summary

- [ ] Summary explains what changed.
- [ ] Purpose explains why the change is needed.
- [ ] Scope and out-of-scope items are clear.
- [ ] Related issue/requirement is linked or marked `Needs confirmation`.

## Implementation

- [ ] Affected modules/files are listed.
- [ ] Implementation approach is summarized.
- [ ] Backward compatibility is addressed.
- [ ] Known limitations are documented.

## Verification

- [ ] Relevant test commands were run and results recorded.
- [ ] Any checks not run are listed with reason.
- [ ] Manual checks are described when relevant.
- [ ] Screenshots/recordings/API examples/logs are attached or referenced when relevant.
- [ ] No claim says a test passed unless it actually ran successfully.

## Risk and Release

- [ ] API contract changes are documented or marked N/A.
- [ ] Config/environment changes are documented or marked N/A.
- [ ] Database/migration changes are documented or marked N/A.
- [ ] Security impact is assessed.
- [ ] Performance impact is assessed.
- [ ] Rollback considerations are documented.
- [ ] Deployment notes are documented or marked N/A.

## QA Handover

- [ ] QA validation scope is clear.
- [ ] Test data/environment is documented or marked `Needs confirmation`.
- [ ] Known risk areas are listed.
- [ ] Release blockers, if any, are called out.

## Documentation

- [ ] User/developer/QA/docs changes are updated or marked N/A.
- [ ] New prompts/templates/standards are linked from the correct index when relevant.
